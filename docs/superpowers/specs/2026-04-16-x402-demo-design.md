# x402 Protocol Interactive Demo вҖ” Design Spec

**Date:** 2026-04-16  
**Status:** Approved

---

## Overview

An educational website where developers walk through the x402 payment protocol HTTP handshake step by step. Each step displays real HTTP messages (headers, JSON response bodies) with Chinese annotation bubbles explaining field meanings. The demo uses real EIP-712/EIP-3009 cryptographic logic вҖ” nothing is faked.

**Target audience:** Developers (global). UI language is English; annotation bubbles explaining protocol details are in Chinese.

## Backend API

### `GET /api/server/weather`

**Without `X-PAYMENT` header:**
Returns HTTP 402 with payment requirement:
```json
{
  "x402Version": 1,
  "accepts": [{
    "scheme": "exact",
    "network": "eip155:8453",
    "maxAmountRequired": "10000",
    "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
    "payTo": "0x70997970C51812dc3A010C7d01b50e0d17dc79C8",
    "resource": "/api/server/weather",
    "description": "Weather data вҖ” $0.01 USDC per request",
    "mimeType": "application/json",
    "maxTimeoutSeconds": 300
  }]
}
```

**With `X-PAYMENT` header:**
1. Base64-decode the header value to get payment payload
2. Call facilitator verify logic
3. Pass вҶ’ return 200 + weather data + `X-Payment-Receipt` header
4. Fail вҶ’ return 402 + `{ "error": "Payment verification failed", "reason": "..." }`

### `POST /api/facilitator/verify`

Accepts payment payload, runs 6 checks in order, returns result with per-check status:

| # | Check | Failure reason |
|---|-------|----------------|
| 1 | Field completeness | `missing fields` |
| 2 | Time validity (`validAfter` вүӨ now < `validBefore`) | `payment expired` or `not yet valid` |
| 3 | Amount (`value` вүҘ 10000) | `insufficient amount` |
| 4 | Recipient (`to` == `DEMO_PAY_TO_ADDRESS`) | `wrong recipient` |
| 5 | EIP-712 signature (recover signer, compare to `from`) | `invalid signature` |
| 6 | Nonce deduplication (in-memory set) | `nonce already used` |

Response format:
```json
{ "valid": true, "checks": [{"name": "...", "passed": true}, ...] }
// or
{ "valid": false, "reason": "nonce already used", "checks": [...] }
```

The `checks` array is always returned so the frontend can display each check with a green tick or red cross.

### `GET /api/demo/reset`

Clears the in-memory nonce set, allowing the user to re-run the full flow.

### CORS

Allowed origins:
- `https://x402.oasaka.xyz` (production)
- `http://localhost:8080` (local development)

## Frontend

### UI Language
- Main UI (buttons, titles, step names, HTTP messages): **English**
- Annotation bubbles explaining protocol fields: **Chinese**

### Layout

Three-column layout representing three roles:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ   CLIENT    в”Ӯ   YOUR SERVER    в”Ӯ   FACILITATOR    в”Ӯ
в”Ӯ  (Browser)  в”Ӯ   (FastAPI)      в”Ӯ  (Mock Verifier) в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

- Desktop: three columns side by side with animated arrows between them
- Mobile: single column, steps shown in order

Top of page: 7-step progress indicator (circles, current step highlighted, completed steps checked).

### HTTP Message Cards

Each HTTP message shown in a "Terminal card":
- Dark background (`#0d1117`), rounded corners, macOS-style decorative dots
- Syntax highlighting: method (blue), 200 (green), 402 (orange), header names (purple), JSON keys (cyan)
- Appear with fade-in + slight upward slide animation
- Chinese annotation bubbles appear alongside to explain field meanings

### 7-Step Interaction Flow

User must click a button to advance each step вҖ” no auto-progression.

| Step | Frontend Action |
|------|----------------|
| 1 | Display GET request card, fetch `/api/server/weather` (no payment header) |
| 2 | Display 402 response, show annotation bubbles field by field |
| 3 | Use ethers.js (CDN) to build EIP-3009 payload; fields appear one by one with animation |
| 4 | Display request card with `X-PAYMENT` header, re-fetch with payment |
| 5 | Display ServerвҶ’Facilitator internal call structure |
| 6 | Display 6 verification checks with animated tick/cross per item |
| 7 | Display 200 response with weather data, play success animation |

After step 7: **"Try Replay Attack"** button appears. Clicking it re-sends the same nonce вҶ’ Facilitator rejects with `"reason": "nonce already used"` вҶ’ frontend displays the rejection, teaching the anti-replay mechanism without any extra explanation.

## Deployment

### Local Development

```bash
# Terminal 1 вҖ” backend
cd backend && uv sync
uv run uvicorn main:app --reload --port 8000

# Terminal 2 вҖ” frontend
cd frontend && python -m http.server 8080
# Open http://localhost:8080
```

### CI/CD (GitHub Actions)

Single workflow file (`.github/workflows/deploy.yml`) with two jobs triggered on push to `main`:

**`deploy-frontend`**
- Publishes `frontend/` to GitHub Pages
- Custom domain: `x402.oasaka.xyz` (configured via `CNAME` file in `frontend/`)

**`deploy-backend`**
- SSHs into VPS using private key stored in GitHub Secrets (`VPS_SSH_KEY`, `VPS_HOST`, `VPS_USER`)
- Runs: `git pull && cd backend && uv sync && systemctl restart x402-backend`

### First-Time Deploy Order
1. VPS: configure Nginx (`nginx/x402.conf`) + SSL via Let's Encrypt for `api.oasaka.xyz`
2. VPS: create systemd service `x402-backend`
3. GitHub: add `VPS_SSH_KEY`, `VPS_HOST`, `VPS_USER` to repo Secrets
4. DNS: add `CNAME x402 вҶ’ ethan0xbuilds.github.io`
5. Push to `main` вҶ’ Actions deploys both frontend and backend

## Implementation Order (Backend-First)

1. **Backend skeleton** вҖ” FastAPI app, CORS, routes wired up, config constants
2. **crypto.py** вҖ” EIP-712 verification logic + Python test script to self-verify a test signature
3. **Facilitator endpoint** вҖ” all 6 checks, verified with curl
4. **Server endpoint** вҖ” 402 response and 200 response with payment verification
5. **Frontend HTML/CSS** вҖ” static layout, terminal cards, step indicator (no JS logic yet)
6. **Frontend JS** вҖ” step-by-step interaction, ethers.js signing, API calls
7. **Replay attack demo** вҖ” add reset endpoint + replay button
8. **GitHub Actions** вҖ” CI/CD workflow for both frontend and backend
9. **Nginx config** вҖ” `api.oasaka.xyz` server block

## Acceptance Criteria

- User can walk through all 7 steps without knowing blockchain, and understand the x402 handshake
- Every step shows real HTTP messages, not fake data
- EIP-712/EIP-3009 signing and verification are real cryptographic operations
- Replay attack demo correctly rejected by Facilitator with `"nonce already used"`
- Frontend deployed to `x402.oasaka.xyz`, backend running on `api.oasaka.xyz`
- Mobile-friendly
